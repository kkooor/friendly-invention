一、运输层概述

## 1. 核心任务

为**应用进程之间**提供**端到端**的逻辑通信服务，屏蔽网络层及以下的具体细节（如路由转发、分片等），使应用层无需关注底层传输细节。

## 2. 在 TCP/IP 协议栈中的位置

运输层位于应用层之下、网络层之上，是端到端通信的核心，核心协议是 **TCP（传输控制协议）** 和 **UDP（用户数据报协议）**。

## 3. 数据单元

运输层协议数据单元（PDU）：**报文段（Segment，TCP 专用）** 或 **用户数据报（User Datagram，UDP 专用）**。

## 4. 关键概念：复用与分用（必考）

| 操作     | 方向   | 说明                                                         |
| :------- | :----- | :----------------------------------------------------------- |
| **复用** | 发送方 | 不同应用进程的报文，封装上各自的端口号，统一交付给网络层（一个网络层IP数据报可承载多个运输层报文）。 |
| **分用** | 接收方 | 根据运输层首部中的**目的端口号**，将数据正确交付给对应应用进程（解封装后分流）。 |

## 5. 端口号分类（高频考点）

| 类型              | 范围        | 说明                                                         |
| :---------------- | :---------- | :----------------------------------------------------------- |
| **知名端口**      | 0~1023      | 由 IANA 分配给固定服务（必背：HTTP=80, DNS=53, FTP=21, SMTP=25, SSH=22, Telnet=23）。 |
| **注册端口**      | 1024~49151  | 可由软件厂商注册使用（如Tomcat默认8080），非固定但可公开查询。 |
| **动态/临时端口** | 49152~65535 | 客户端进程临时使用，通信结束后释放，无需注册。               |

易错点：端口号是**应用进程的标识**，不是主机的标识；一个主机上可同时运行多个应用进程，每个进程对应唯一端口号。

# 二、UDP（用户数据报协议）

## 1. 核心特点（必考）

- **无连接**：发送数据前不需要建立连接，直接发送，减少开销。
- **不可靠交付**：不保证数据到达、不保证顺序、无拥塞控制，丢包不重传。
- **面向报文**：保留应用层报文边界，一次交付一个完整报文，不拆分、不合并（应用层发多少，UDP就传多少）。
- **开销小**：首部仅 8 字节，远小于TCP。
- **支持组播和广播**：TCP 仅支持单播，UDP 可用于多终端通信（如直播）。

## 2. 应用场景（对应特点记忆）

实时性优先、可容忍少量丢包的场景：实时音视频流（直播、 VoIP）、在线游戏、DNS 查询、SNMP（简单网络管理协议）、RIP 路由协议、TFTP（简单文件传输协议）。

## 3. UDP 首部格式（高频考点）

| 字段       | 长度   | 说明                                                         |
| :--------- | :----- | :----------------------------------------------------------- |
| 源端口号   | 16 bit | 发送进程的端口号，不需要回复时可全为 0。                     |
| 目的端口号 | 16 bit | 接收进程的端口号，核心字段（用于分用）。                     |
| 长度       | 16 bit | UDP 首部 + 数据的总长度（以字节为单位），最小 8 字节（仅首部）。 |
| 检验和     | 16 bit | 检验整个 UDP 数据报，**IPv4 中可选，IPv6 中强制**；计算覆盖伪首部 + UDP 首部 + 数据。 |

补充：伪首部（仅用于检验和计算，不实际传输）包含 4 个字段：源 IP 地址、目的 IP 地址、协议号（UDP=17）、UDP 总长度，目的是确保数据交付到正确的主机和协议。

# 三、TCP（传输控制协议）概述

## 1. TCP 的核心特点（必考，与UDP对比记忆）

- **面向连接**：通信前必须通过三次握手建立连接，通信结束后通过四次挥手释放连接。
- **全双工通信**：连接双方可同时收发数据，各自拥有发送和接收缓冲区。
- **可靠有序**：通过序号、确认、重传等机制，保证数据无差错、不丢失、不重复、按序到达。
- **面向字节流**：将应用层报文看作无结构的字节流，不保留报文边界（可拆分、合并）。
- **支持流量控制与拥塞控制**：避免接收方缓冲区溢出、避免网络过载。

## 2. TCP 连接（核心概念）

- 每条 TCP 连接有且仅有两个端点，端点称为**套接字（Socket）**，格式为：**（IP 地址 : 端口号）**。
- TCP 连接的唯一标识：`TCP 连接 = {源套接字，目的套接字}`，即（源IP:源端口，目的IP:目的端口），同一对套接字不能同时建立多个连接。

# 四、TCP 报文段格式（必考重点）

| 字段              | 长度   | 核心说明                                                     |
| :---------------- | :----- | :----------------------------------------------------------- |
| 源端口号          | 16 bit | 发送方应用进程端口                                           |
| 目的端口号        | 16 bit | 接收方应用进程端口，用于分用                                 |
| **序号（seq）**   | 32 bit | 本报文段数据**第一个字节**的序号（面向字节流，每个字节都有唯一序号）。 |
| **确认号（ack）** | 32 bit | 期望收到对方下一个报文段的第一个数据字节序号（**累积确认**），ack = 已正确接收的最后一个字节序号 + 1。 |
| 数据偏移          | 4 bit  | 首部长度（以 4 字节为单位），最小值为 5（对应 20 字节固定首部），最大值为 15（对应 60 字节首部）。 |
| 保留              | 6 bit  | 预留，暂置 0，未使用                                         |
| **控制位**        | 6 bit  | URG, ACK, PSH, RST, SYN, FIN（关键，必考）                   |
| 窗口大小          | 16 bit | 接收方当前接收缓冲区的可用空间，用于**流量控制**，告知发送方可发送的最大字节数。 |
| 检验和            | 16 bit | 计算覆盖伪首部 + TCP 首部 + 数据，检测比特差错。             |
| 紧急指针          | 16 bit | 当 URG=1 时有效，标识紧急数据的末尾偏移（紧急数据优先交付）。 |
| 选项              | 可变   | 常用：最大报文段长度（MSS，默认 1460 字节）、窗口扩大因子、时间戳（用于计算 RTT）、选择确认（SACK）。 |

## 关键控制位说明（必考，记清含义和用途）

| 标志位  | 含义           | 核心用途                                                     |
| :------ | :------------- | :----------------------------------------------------------- |
| **SYN** | 同步序号       | 连接建立时使用，用于同步双方的初始序号（SYN=1 表示建立连接请求）。 |
| **FIN** | 释放连接       | 通信结束时使用，表示发送方已无数据要发送，请求释放连接。     |
| **ACK** | 确认号字段有效 | 连接建立后通常始终置 1，标识确认号有效，用于可靠传输的确认。 |
| **RST** | 重置连接       | 异常关闭连接（如连接出错、端口未监听），强制终止连接。       |
| **PSH** | 推送           | 提示接收方尽快将缓冲区中的数据向上交付给应用层，不等待缓冲区满（如实时交互场景）。 |
| **URG** | 紧急指针有效   | 标识报文段中包含紧急数据，紧急指针指向紧急数据的末尾。       |

# 五、TCP 可靠传输的实现（必考重点）

## 1. 停止等待协议（Stop-and-Wait）

- 核心逻辑：每发送一个报文段，就停止发送，等待接收方的确认（ACK），超时未收到则重传。
- 优点：简单、易实现。
- 缺点：**信道利用率极低**（尤其是长距离、高带宽网络），仅适用于低速、低延迟网络。

## 2. 滑动窗口协议（连续 ARQ + 累积确认）

TCP 实际采用的可靠传输机制，结合**连续 ARQ 协议**和**累积确认**，实现高效可靠传输：

- 发送方：在窗口范围内，可连续发送多个报文段，无需等待每个报文段的确认。
- 接收方：无需逐个确认，收到多个报文段后，发送一个累积确认（确认号为已正确接收的最后一个字节序号 + 1）。

## 3. 关键可靠机制（必考，记清作用）

| 机制              | 核心作用                                         | 补充说明                                                     |
| :---------------- | :----------------------------------------------- | :----------------------------------------------------------- |
| **序号与确认**    | 保证数据按序重组，确认号表明期望接收的下一个字节 | 累积确认：ack = N 表示 N-1 及之前所有字节已正确收到，中间有缺失则确认号不跳过。 |
| **超时重传**      | 应对报文段丢失，超时则重传                       | 超时时间 RTO 由往返时间 RTT 自适应估算（Jacobson/Karels 算法），动态调整，避免过早/过晚重传。 |
| **快速重传**      | 快速检测报文段丢失，减少重传延迟                 | 收到 **3 个重复 ACK** 时，不等超时，立即重传缺失的报文段（判断为报文段丢失，而非超时）。 |
| **选择确认 SACK** | 减少不必要的重传，提高效率                       | 接收方通知发送方已收到的不连续字节块，发送方只重传确实丢失的段（需开启 TCP 选项 SACK）。 |
| **校验和**        | 检测比特差错（如传输过程中的误码）               | 校验范围包含伪首部，确保数据交付到正确的主机和协议。         |

易错点：累积确认的核心是“确认已收到的所有连续字节”，若中间有报文段丢失，接收方会持续发送重复的确认号（即重复 ACK），触发快速重传。

# 六、TCP 流量控制

## 1. 目的

防止发送方发送速率过快，超过接收方的接收能力，导致接收缓冲区溢出，数据丢失。（核心：保护接收方）

## 2. 实现机制：滑动窗口协议

- 接收方：在 TCP 首部的**窗口大小**字段，告知发送方自己当前接收缓冲区的可用空间（以字节为单位），称为**接收窗口（rwnd）**。
- 发送方：发送窗口的大小由接收窗口和拥塞窗口共同决定，即：$$发送窗口 = \min(rwnd,\ cwnd)$$（cwnd 为拥塞窗口，后续讲解）。
- 特殊情况：当接收方缓冲区满（rwnd = 0）时，发送方停止发送数据，并定期发送**零窗口探测报文**，查询接收方缓冲区是否有空闲空间，避免死锁。

补充：流量控制是“端到端”的控制，仅关注发送方和接收方的能力，与网络无关。

# 七、TCP 拥塞控制（必考难点）

## 1. 拥塞控制与流量控制的区别（必考对比）

| 比较项   | 流量控制                            | 拥塞控制                                         |
| :------- | :---------------------------------- | :----------------------------------------------- |
| 核心目的 | 保护**接收方**缓冲区不溢出          | 保护**网络**不发生过载（避免路由器队列满、丢包） |
| 关注点   | 端到端的接收能力（发送方 ↔ 接收方） | 全网结点的承载能力（路由器、链路等）             |
| 控制依据 | 接收窗口（rwnd）                    | 拥塞窗口（cwnd）                                 |

## 2. TCP 拥塞控制算法（四大核心机制，必考流程）

| 机制         | 核心说明                                                     | cwnd 变化规律                                                |
| :----------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **慢启动**   | 连接建立初期，避免突然发送大量数据导致网络拥塞，逐步增加发送速率。 | cwnd 初始设为 1 MSS（或 10 MSS），每收到一个 ACK，cwnd 翻倍（指数增长），直到达到慢启动阈值 ssthresh（初始 ssthresh 为较大值，如 65535 字节）。 |
| **拥塞避免** | cwnd 达到 ssthresh 后，减缓增长速率，避免网络拥塞。          | cwnd ≥ ssthresh 时，每经过一个 RTT（往返时间），cwnd 增加 1 MSS（线性增长）。 |
| **快重传**   | 快速检测报文段丢失，无需等待超时，减少重传延迟。             | 收到 3 个重复 ACK 时，立即重传丢失的报文段，触发快恢复。     |
| **快恢复**   | 重传后，快速调整 cwnd 和 ssthresh，避免再次拥塞，直接进入拥塞避免阶段。 | 将 ssthresh 设为当前 cwnd 的一半，cwnd 也设为 ssthresh（或 ssthresh + 3 MSS），然后进入拥塞避免，线性增长。 |

## 3. 拥塞判定与处理（必考）

- **超时重传（严重拥塞）**：         判定：发送方超时未收到 ACK，认为网络严重拥塞。         处理：ssthresh = cwnd / 2，cwnd = 1 MSS，重新进入慢启动阶段。
- **重复 ACK（轻微拥塞）**：         判定：收到 3 个重复 ACK，认为个别报文段丢失，网络轻微拥塞。         处理：执行快重传 + 快恢复，不重新慢启动，直接进入拥塞避免。

# 八、TCP 连接管理（三次握手+四次挥手，必考重中之重）

## 1. 三次握手（建立连接，核心流程）

目的：建立连接，同步双方的初始序号，确认彼此的收发能力。

| 步骤 | 方向                     | 报文类型       | 关键内容（seq/ack/控制位）                           | 核心目的                                                     |
| :--- | :----------------------- | :------------- | :--------------------------------------------------- | :----------------------------------------------------------- |
| 1    | A（客户端）→ B（服务器） | SYN 报文       | seq = x（随机初始序号），SYN=1，ACK=0                | A 向 B 发起连接请求，同步自己的初始序号 x。                  |
| 2    | B → A                    | SYN + ACK 报文 | seq = y（B 的随机初始序号），ack = x+1，SYN=1, ACK=1 | B 确认收到 A 的请求，同步自己的初始序号 y，告知 A“已收到 x 及之前的所有字节”。 |
| 3    | A → B                    | ACK 报文       | seq = x+1，ack = y+1，ACK=1，SYN=0                   | A 确认收到 B 的同步，告知 B“已收到 y 及之前的所有字节”，连接建立完成。 |

## 2. 四次挥手（释放连接，核心流程）

目的：释放连接，确保双方都已完成数据传输，无残留数据。（TCP 全双工，需双方分别释放）

| 步骤 | 方向  | 报文类型 | 关键内容（seq/ack/控制位）                          | 核心目的                                                     |
| :--- | :---- | :------- | :-------------------------------------------------- | :----------------------------------------------------------- |
| 1    | A → B | FIN 报文 | seq = u（A 已发送的最后一个字节序号），FIN=1，ACK=1 | A 告知 B：“我已无数据要发送，请求释放我到 B 的连接”。        |
| 2    | B → A | ACK 报文 | ack = u+1，seq = v，ACK=1，FIN=0                    | B 确认收到 A 的释放请求，此时 B 仍可向 A 发送数据（全双工）。 |
| 3    | B → A | FIN 报文 | seq = v（B 已发送的最后一个字节序号），FIN=1，ACK=1 | B 告知 A：“我也无数据要发送，请求释放 B 到 A 的连接”。       |
| 4    | A → B | ACK 报文 | ack = v+1，seq = u+1，ACK=1，FIN=0                  | A 确认收到 B 的释放请求，告知 B 可以释放连接。               |

## 3. 关键问题（必考简答题）

- **为什么是三次握手而不是两次？** 答：防止“已失效的连接请求报文段”突然传到服务器，导致错误。例如：A 发送的连接请求超时未被 B 收到，A 重新发送请求并建立连接、通信结束；若此时旧的请求报文段到达 B，B 会误以为 A 又发起新连接，若仅两次握手，B 会直接建立连接，而 A 不会响应，导致 B 资源浪费。三次握手可让 A 确认 B 收到了自己的正常请求，避免该问题。      
- **为什么需要 TIME-WAIT 状态？** 答：TIME-WAIT 状态是 A 发送最后一个 ACK 后进入的状态，等待 **2MSL（最长报文段寿命）**，目的有两个：         ① 保证最后一个 ACK 能被 B 收到（若 B 未收到 ACK，会重发 FIN，A 在 TIME-WAIT 期间可重新发送 ACK）；         ② 让网络中残留的旧连接报文段（如重复的 FIN、ACK）消失，避免其干扰新连接。      
- **为什么四次挥手而不是三次？** 答：TCP 是全双工通信，双方都有发送数据的权利。当 A 发送 FIN 释放自己的发送端后，B 可能还在向 A 发送数据，无法立即发送 FIN，需先发送 ACK 确认，待 B 发送完所有数据后，再发送 FIN 释放自己的发送端，因此需要四次挥手。      

# 九、TCP 与 UDP 对比（必考，表格记忆）

| 特性              | TCP（传输控制协议）                            | UDP（用户数据报协议）                                 |
| :---------------- | :--------------------------------------------- | :---------------------------------------------------- |
| 连接方式          | 面向连接（三次握手建立，四次挥手释放）         | 无连接（直接发送，无需建立/释放）                     |
| 可靠性            | 可靠，保证数据无差错、不丢失、不重复、按序到达 | 不可靠，尽最大努力交付，丢包不重传                    |
| 数据边界          | 面向字节流，无边界（可拆分、合并）             | 面向报文，保留应用层报文边界                          |
| 流量控制/拥塞控制 | 支持（滑动窗口、拥塞控制算法）                 | 不支持                                                |
| 首部开销          | 较大，固定 20~60 字节                          | 较小，固定 8 字节                                     |
| 通信方式          | 仅支持单播                                     | 支持单播、组播、广播                                  |
| 典型应用          | HTTP、FTP、SMTP、SSH、Telnet（对可靠性要求高） | DNS、实时视频、VoIP、在线游戏、TFTP（对实时性要求高） |

```plain
运输层
├─ 概述
│  ├─ 核心任务：端到端应用进程逻辑通信
│  ├─ 数据单元：TCP报文段、UDP用户数据报
│  ├─ 关键机制：复用（发送方）、分用（接收方）

├─ UDP协议
│  ├─ 特点：无连接、不可靠、面向报文、开销小、支持组播广播

│  └─ 应用场景：实时音视频、游戏、DNS等
├─ TCP协议
│  ├─ 特点：面向连接、可靠、全双工、面向字节流、支持流量/拥塞控制
│  ├─ 报文段格式：核心字段（seq、ack、控制位、窗口大小）
│  ├─ 可靠传输：停止等待、滑动窗口（连续ARQ+累积确认）、超时重传、快速重传等
│  ├─ 流量控制：基于接收窗口（rwnd）的滑动窗口协议s
│  ├─ 拥塞控制：慢启动、拥塞避免、快重传、快恢复（cwnd+ssthresh）
│  └─ 连接管理：三次握手（建立）、四次挥手（释放）、TIME-WAIT状态
├─ TCP与UDP对比：核心特性、应用场景差异
└─ 高频考点：计算（序号、拥塞控制）、简答（三次握手、四次挥手等）
```
